01

What Does the Recommendation Hub Do?

In a typical distribution sales operation, a rep walks into a partner's shop with either a preset target ("sell 50 units of X this month") or no structure at all. Neither works well. The Recommendation Hub gives each rep a ranked, personalised action plan for every partner — backed by that partner's actual transaction data.

Here is what happens end-to-end in one session:

1
Signal Collection
The engine reads partner health, churn probability, credit risk, revenue gaps vs cluster peers, and association rule matches for this specific partner.
2
Action Generation
Based on which signals fire, the engine creates up to 6 types of recommended actions — cross-sell, up-sell, retention, affinity bundle, credit-safe, or strategic expansion.
3
Scoring & Ranking
Each action gets a priority score (0–100). Actions are deduplicated (highest score per type) and ranked. The top action becomes the "primary pitch."
4
Pricing & Elasticity
For the top action, the engine looks up the product's real transfer price, computes the safe discount ceiling using Price Elasticity of Demand, and generates a margin-protected offer line.
5
Script Generation
Produces a ready-to-send WhatsApp message and a structured email — personalised to the partner's name, the recommended product, price, and tone (Friendly / Formal / Urgent).
6
Outcome Recording
The rep logs what happened — accepted, rejected, won, or lost. This feeds the weekly learning summary, which shows which action types and tones are actually converting.
02

Recommendation Scoring Pipeline

recommendation_mixin.py → _build_partner_actions()

Every recommendation starts as a "raw signal" that fires when a condition is met. The signal is converted into an action type, and then scored. Here is the full scoring table:

Signal ConditionAction TypeBase Priority ScoreModifiers
Peer gap ≥ ₹1 across any product Cross-sell / Up-sell 55 + gap_size × 0.00025 (bigger gap = higher priority)
FP-Growth rule: confidence ≥ 0.10 or lift ≥ 1.1 Affinity Bundle 58 + (confidence × 30) + (2 if lift ≥ 1.5)
Health segment "At Risk" or churn probability ≥ 45% Retention Intervention 70 + (estimated monthly loss / 100,000)
Credit risk score ≥ 55% Credit-Safe Action 65 + (credit_score × 40)
VIP tier + Champion or Healthy segment Strategic Expansion 52–60 + VIP-specific bonus
Any active alert on the partner Alert-Led Escalation 75 + severity multiplier from alert metadata

Why These Base Scores?

The base scores are designed so that revenue protection (retention = 70, alerts = 75) always outranks revenue growth (cross-sell = 55, affinity = 58) by default. The logic is simple: protecting existing revenue is always more urgent than growing it. A rep facing a churning partner should not be told to cross-sell — they should be told to rescue the account first.

Strategic expansion gets the lowest base (52) because it's only relevant when a partner is already healthy. Suggesting expansion to a declining partner would be tone-deaf.

Deduplication and Ranking

For each action_type, keep only the highest-scoring action Then sort all remaining actions descending by priority_score If query contains "margin" → sort by: margin_rate first, then priority_score Otherwise → sort by: priority_score first, then margin_rate

This design means if a partner has three cross-sell opportunities (mobiles, accessories, peripherals), the system shows only the best-scoring one as the primary action — keeping the output clean and actionable. The others surface as alternates during follow-up.

🔍
Why not just use a machine learning ranker? A learning-to-rank model (like LambdaMART or pairwise XGBoost ranker) would learn optimal ordering from labelled outcome data. That is absolutely the right long-term direction — and the feedback table is already in place to collect that data. Right now there are not enough labelled conversions (accepted/won/rejected per recommendation) to train a reliable ranker. The rule-based scoring is a calibrated interim solution that the team can reason about and correct when wrong.
03

Confidence %, Similar Partners & Expected Uplift

recommendation_mixin.py → _build_partner_actions()

Every recommendation now surfaces three business-facing signals designed to answer the rep's unspoken question: "Should I really trust this?"

Confidence Percentage

confidence_pct = 50 + (priority_score ÷ 100) × 47 Maps priority scores to the 50–97% range, never implying 100% certainty. Examples: priority = 55 → confidence = 75.9% priority = 70 → confidence = 82.9% priority = 90 → confidence = 92.3%

The 50% floor matters — it means the system never shows a recommendation with less than 50% confidence. If a signal isn't strong enough to pass 50% confidence, it doesn't appear at all (it gets filtered out by the minimum priority threshold before this calculation).

Similar Partners Count (Peer Validation)

n_cluster_peers = count of partners in same cluster with same cluster_label For Retention / Credit actions (typically affects a subset of cluster): similar_partners_count = n_cluster_peers × 0.40 For Cross-sell / Affinity actions (gap exists for most healthy peers): similar_partners_count = n_cluster_peers × 0.75

This number answers the question: "Is this just about this one partner or is it a pattern?" If 18 similar partners are showing the same gap, the rep can truthfully tell the partner: "We see this with a lot of distributors like you in this region."

Expected Uplift per Month

Priority 1: Extract monetary amount from recommendation text regex match: "estimated monthly upside Rs X" or "₹X/month" Priority 2: Fallback formula if no Rs amount in recommendation text expected_uplift_monthly = (recent_90d_revenue ÷ 3) × (priority_score ÷ 100) × 0.15 Interpretation: "15% of monthly revenue × confidence" — a conservative floor estimate. This is intentionally conservative. We would rather under-promise than over-promise.

The 15% multiplier is the key design decision here. We could have used 30% or 50%, which would produce flashier numbers. We chose 15% because in our product category (B2B distribution), a 15% uplift from adding a new product category is achievable and verified — while a 50% uplift would set expectations that damage credibility when they don't materialise.

💡
Design principle: GPT patches, it does not replace. The heuristic parser runs first. If it successfully extracts all filter values, GPT is not called even if the API key exists. GPT is only invoked to fill gaps the heuristic missed — this keeps API costs low and ensures the system works fully offline without any API key.

Scanning Limit

NL_QUERY_PARTNER_SCAN_LIMIT = 300 (configurable via .env) The engine scans up to 300 partners per query, filtering by the parsed criteria. Prevents slow queries on very large partner bases while keeping results accurate. If state filter is applied, the scan is restricted to that state first.

Margin-Safe Pricing Logic

recommendation_mixin.py → _lookup_offer_pricing()

Before the system generates a pitch, it looks up the actual transfer price and selling price of the recommended product from the database. It then calculates how much discount can be safely offered without hurting the company's margin.

Price Lookup Query

The system queries two sources and picks the best available:

SourceHowFallback
Transaction averageavg(net_amt / qty) over last 180 days for this productIf no transactions, use transfer price × 1.18 (GST markup)
Transfer pricemaster_product_selling_price — latest record per productIf neither available, unit_price = NaN (shown as "as per rate card")
Margin rate(net_amt − transfer_price × qty) / net_amt averagedDefault = 15% if not calculable

Margin rate is clamped between 2% and 60% — below 2% is probably a data error; above 60% is implausible for B2B distribution and was causing anomalous discount calculations in early testing.

06

Price Elasticity of Demand (PED)

recommendation_mixin.py → _lookup_offer_pricing() + mv_product_elasticity_stats

This is one of the more sophisticated components. Instead of giving every partner the same discount, the system calculates whether this specific product's demand is elastic (discounts drive volume) or inelastic (demand doesn't change much with price). Different strategies apply in each case.

What is Price Elasticity?

Elasticity measures how much quantity demanded changes when price changes. In economics:

PED = % Change in Quantity Demanded ÷ % Change in Price PED > 1 → Elastic (small price drop → big volume increase) PED < 1 → Inelastic (price drops don't move the needle much) PED ≈ 1 → Unit elastic (proportional response)

How We Approximate It (Without Price Experiments)

We don't run price experiments. Instead, we use Coefficient of Variation (CV) as a proxy stored in a materialised view (mv_product_elasticity_stats):

volume_CV = std(monthly_quantity) ÷ mean(monthly_quantity) ← How volatile is demand? price_CV = std(unit_price) ÷ mean(unit_price) ← How much does price vary? elasticity_score = volume_CV ÷ price_CV (If price barely varies but volume swings a lot → highly elastic demand proxy)

Discount Strategy Based on Elasticity

Elasticity ScoreSignalStrategySafe Discount
> 1.2 Highly Elastic Demand spikes with price drops. Offer bigger discount to drive volume. Up to 70% of margin (max 15%)
0.1 – 0.8 Inelastic Partner buys regardless of price. Don't give away margin unnecessarily. 2% (relationship gesture only)
0.8 – 1.2 Neutral Standard pricing. Moderate discount to incentivise the action. Up to 40% of margin (max 8%)
⚖️
Limitation — CV-based elasticity is an approximation True PED requires controlled price experiments, which we cannot run at scale in a distribution business. The CV proxy works well when price varies naturally (different distributors get different rates, seasonal pricing) but breaks down for products where price is always fixed. For fixed-price products, the elasticity score defaults to 1.0 (neutral) and the standard 40%-of-margin discount applies. This is a conservative default — it neither over-discounts nor under-discounts in the absence of data.

Pitch Script Generation

recommendation_mixin.py → _build_pitch_templates()

The system generates ready-to-send communication for the rep — both WhatsApp and email — personalised to the partner's name, the recommended product, the safe price, and the selected tone. All of this happens in Python without any API call, so it's instant and free.

Three Tones Available

ToneWhatsApp OpeningEmail CTA
Friendly"Hi [Name], quick suggestion for your next order.""Reply with quantity and I will block stock today."
Formal"Dear [Name], recommendation based on your current portfolio.""Please confirm your preferred quantity and schedule."
Urgent"Hi [Name], urgent opportunity for this cycle.""Please confirm today so we can lock this offer."

What Goes Into Every Script

WhatsApp format (short and scannable): {opening based on tone} Recommended: {product_name} ({action_type}) Indicative unit price: Rs {unit_price:,} Margin-safe offer: up to {discount}% off (~ Rs {offer_price:,}/unit) Protected margin band: ~{margin %}% {CTA based on tone} Email format (structured and professional): Subject: {product_name} recommendation for your next cycle Body: Reason (why_relevant from the action) + price + offer + margin + CTA

GenAI Enhancement (Optional)

If OPENAI_API_KEY is provided, the system sends the deterministic script to GPT as a "baseline" and asks it to produce a more natural-sounding version. The prompt explicitly tells GPT the tone, channel requirements, and that it must include product, price, margin offer, and CTA. If the API is unavailable, the deterministic script is used — the user never sees a blank or broken output.

ℹ️
Why generate deterministic scripts at all, if GenAI can do it better? Three reasons: (1) GenAI costs money per call. A field force of 100 reps generating 10 scripts each per day = 1,000 API calls. (2) GenAI can hallucinate prices or discount percentages — the deterministic script is always mathematically correct. (3) The deterministic version works offline, on slow connections, or when the API key is not configured. GenAI is an enhancement, not a dependency.

Follow-Up Sequencing

recommendation_mixin.py → get_partner_followup_scripts()

A recommendation that wasn't acted on in 7 days needs a different message — not the same script again. The follow-up system generates a new communication that acknowledges the delay, lowers the friction (smaller trial quantity), and offers an alternate product if the primary one wasn't appealing.

How It Works

1
Pull original pitch data
The system re-fetches the same recommendation action and pricing that was used in the initial pitch.
2
Find an alternate offer
Checks FP-Growth bundle recommendations first. If none found, falls back to the partner's peer gap analysis to suggest the next-highest-gap product.
3
Build a lower-friction message
Acknowledges that X days have passed. Reduces the ask — suggests a trial quantity (default 5 units). Includes the revised opening hook appropriate to the selected tone.
Follow-up WA format example (Friendly tone, 7 days, 5 units): "Hi [Name], just checking back after 7 days on the previous suggestion." "Since there has been no conversion yet, we are proposing a lower-friction option." "Current offer for [Product]: up to [X]% off (~Rs [price]/unit)" "Alternate bundle option: [Alternate Product]" "Margin-safe band maintained at ~[margin]%" "If useful, we can start with a small trial of 5 units this week."

Feedback & Learning Loop

recommendation_mixin.py → record_recommendation_feedback() + get_weekly_feedback_learning_summary()

This is the part that makes the system get smarter over time — if we use it. Every time a rep marks a recommendation as accepted, rejected, won, or lost, that outcome is stored in the database alongside all the context (which action type, what tone, what channel, what the partner's churn and credit scores were at the time).

Feedback Table Schema

FieldWhat it captures
outcomeaccepted / rejected / won / lost
stageinitial_pitch or followup
channelwhatsapp / email / call / in_person
tonefriendly / formal / urgent
action_typecross-sell / retention / affinity / etc.
priority_scoreThe rank score at time of recommendation
churn_probabilityPartner's churn risk at time of pitch
credit_risk_scorePartner's credit risk at time of pitch

Weekly Learning Summary

The system aggregates all feedback from a rolling window (default: last 7 days) and computes:

win_rate = count(outcome == "won") ÷ total_events positive_rate = count(outcome ∈ {accepted, won}) ÷ total_events Grouped by: action_type × (tone × channel) Output: ranked table showing which combinations convert best Example insight: "Friendly tone + WhatsApp + Affinity Bundle = 67% win rate" Example insight: "Urgent tone + Email + Credit-Safe = 12% win rate (lowest)"
🚀
This is the foundation of future reinforcement learning The table schema, the feedback events, and the win-rate aggregation are all in place. The next step is to use these win rates to automatically boost priority scores for action types and tones that are converting well. That is a Thompson Sampling bandit — a well-established technique used by Google and Amazon for recommendation optimisation. The data infrastructure is already built.
