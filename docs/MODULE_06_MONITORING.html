A prediction system that cannot check its own accuracy is a liability, not an asset. The Monitoring module is the self-audit layer of the Sales Intelligence Dashboard. It watches for anomalies in model outputs, validates that the clustering algorithm is producing business-meaningful groups, backtests the degrowth signal against historical outcomes, and surfaces immediate one-liner operational alerts when something needs urgent attention. Think of it as the engine's own health check — running alongside everything else every time the dashboard loads.

Module Architecture — Four Pillars

The Monitoring tab brings together four independent monitoring streams. Each runs on separate data, catches different types of problems, and feeds into the same central dashboard view for the operations team.

🚨
Operational Alerts
Real-time flags when a partner's churn, credit, or revenue metrics cross danger thresholds. Sorted by severity.
🔬
Cluster Quality
Silhouette scores, entropy, stability ARI, outlier ratios, and drift detection — all automated each run.
📋
Degrowth Backtest
Precision/recall evaluation of the degrowth alerting rule against what actually happened in subsequent windows.
🗄️
Data Quality
Missing columns, null ratios, zero-variance features, and critical field health checks before any ML runs.
🎯
Why build monitoring into the same dashboard instead of a separate tool? Separate monitoring dashboards are almost always ignored. By embedding it as a tab within the same tool that sales managers and analysts use daily, it becomes part of the natural workflow. When a senior manager sees an alert on the Monitoring tab while reviewing their Kanban pipeline, they act on it. When it's in a separate Grafana dashboard that nobody checks, it doesn't.

Operational Alerts Engine

monitoring_mixin.py → get_alert_snapshot()

The alert engine checks every partner in the partner features matrix against three independent rule conditions. A partner is flagged if any one of these conditions is true. The more conditions that trigger simultaneously, the higher the alert severity.

⬇ Sharp Revenue Drop
revenue_drop_pct
≥ max(degrowth_threshold, 35%)
Default threshold: 35% drop vs prior period
↑ High Churn Jump
live_churn > batch_churn + 0.15
AND batch_churn ≥ 0.45
Requires realtime scoring to be active
⚠ High Credit Risk Jump
live_credit > batch_credit + 0.15
AND batch_credit ≥ 0.55
Requires realtime scoring to be active

How the Three Rules Were Designed

RuleWhy this structure?Why these thresholds?
Sharp Revenue Drop Compares revenue_drop_pct to the maximum of the partner's personalised degrowth threshold or the global floor. Partners with volatile revenue naturally have higher personal thresholds — prevents false alerts on seasonally cyclical partners. 35% was chosen as the global floor because at this level, a partner's revenue decline is unlikely to be explained by seasonal cycles alone. Below 20%, too many seasonal partners trigger falsely.
High Churn Jump Two-condition gate: the jump (delta between live score and batch score) must be large, AND the absolute churn level must be high. Jump alone is insufficient — a partner going from 0.05 to 0.20 churn is not urgent. But going from 0.40 to 0.55 is. 0.15 jump delta and 0.45 level align with the system's CHURN_PROB_MEDIUM threshold, ensuring alerts fire in the same probability band where the Kanban pipeline starts showing "High Churn" cards.
High Credit Risk Jump Same two-condition structure as churn. Credit risk uses a higher level threshold (0.55 vs 0.45) because false positives in credit risk have commercial consequences — triggering a credit review for a partner wrongly can damage the relationship. 0.55 is the CREDIT_HIGH band entry point. Below this, a partner is still "Medium" risk and doesn't require urgent escalation.

Alert Severity Scoring

monitoring_mixin.py → lines 235–239

When multiple alert conditions trigger for the same partner, the system computes a severity score to rank which partners need attention first. Revenue drops are weighted more heavily than the others because they are immediately visible in financial reports, actionable by the sales team, and a leading indicator of everything else deteriorating.

alert_severity_score = 3 × alert_sharp_revenue_drop + 2 × alert_high_churn_jump + 2 × alert_high_credit_risk_jump Possible scores: 7 = Revenue Drop + Churn Jump + Credit Jump (all 3 — emergency) 5 = Revenue Drop + Churn Jump OR Credit Jump 4 = Churn Jump + Credit Jump (no revenue drop yet) 3 = Revenue Drop only 2 = Churn Jump OR Credit Jump only Table sorted by: severity_score DESC → active_alerts DESC → revenue_drop_pct DESC → churn DESC Partners with the same score are sorted by revenue drop magnitude.

Data Quality Checks

monitoring_mixin.py → get_data_quality_report()

Before any ML model runs, the engine validates the incoming data. This prevents garbage-in-garbage-out scenarios where a missing column silently produces incorrect scores rather than an error. The quality check produces both warnings (non-blocking) and errors (blocking — shown prominently).

CheckWhat's validatedImpact if failed
Missing critical columnsCompany name, revenue, date, product ID — the minimum set required for any feature to be computedBlocking error. The corresponding module feature returns empty data.
Null ratios% of null values in each feature column. null_ratio_max shown in cluster feature reportHigh nullity features are dropped before clustering runs. Shown as warning.
Zero-variance featuresFeatures where all partners have the same value (e.g., every partner has the same cluster assignment)Zero-variance features are pruned — they add no information to ML models. Count shown as zero_var_dropped_count.
Row countTotal rows in the loaded dataset — very low row count may indicate a query filter is too restrictiveWarning if below expected minimum for clustering to be meaningful.
Feature driftmean_abs_z_shift: average Z-score shift of features vs prior run. High drift = distribution changed significantly since last cache.If drift > threshold, clustering is rerun from scratch rather than serving cached labels.
⚠️
The system health banner — first thing you see The top of the Monitoring tab shows a coloured banner: green if no issues, amber if any data quality issues or missing columns are detected. This banner is rendered before any computation results are shown, so a manager know immediately whether to trust the numbers on screen or investigate the data pipeline first.

Cluster Quality Report

clustering_mixin.py → get_cluster_quality_report()

After every clustering run, the system automatically computes a quality report with six metrics. These are shown in the Monitoring tab and used internally to decide whether to accept the clustering result or revert to a fallback.

MetricFormula / meaningGood rangeProblem if…
Silhouette Score Average (intra-cluster cohesion − nearest-cluster distance) / max of both. Ranges −1 to +1. ≥ 0.40 < 0.20: clusters are overlapping — segments aren't meaningful
Calinski-Harabasz Ratio of between-cluster variance to within-cluster variance. Higher is better, no fixed scale. Higher than random baseline If lower than a single-cluster baseline — no structure found
Stability ARI Adjusted Rand Index between two runs with different random seeds. Measures label consistency. ≥ 0.70 < 0.50: small changes in data would radically change who's in which cluster
Outlier Ratio % of partners labelled as Outlier (HDBSCAN noise points). Shown as a KPI metric. ≤ 10% > 25%: HDBSCAN parameters too strict — increase min_cluster_size
Cluster Entropy Shannon entropy of the cluster size distribution. Low = uneven (one huge cluster), High = even split. Depends on cluster count Very low entropy means most partners lumped in one cluster — poor discrimination
Feature Drift Mean absolute Z-score shift of feature distributions vs the prior cached run. < 0.30 > 0.50: data distribution shifted enough that cached cluster labels are stale

Quality Gate & Fallback Logic

Cluster Run
KMeans (VIP tier) + HDBSCAN (Growth tier) execute with contest winners
Quality Check
Silhouette, ARI, outlier ratio evaluated. All must pass thresholds.
Gate: Approved
New labels applied. Cluster metadata updated across all modules.
Gate: Rejected
Fallback applied: prior cached labels retained OR revenue-percentile bucketing used.

The quality gate reason and fallback status are both visible in the Monitoring tab under "Quality Gate and Fallback". This is how an operator knows whether today's cluster labels are freshly computed or are from the last successful run.

Even if a clustering result passes all statistical quality checks, it might still be useless for the sales team if the clusters don't align with business reality. The business validation report compares two approaches against the cluster-guided strategy using three "pitching" simulations:

StrategyHow partners are selected for pitchingPurpose
Cluster-GuidedPrioritise partners with highest churn risk AND positive revenue trend within each cluster — find "at risk but saveable" partnersThe model's actual recommendation strategy
Top-Revenue BaselineJust pitch to the N highest-revenue partners regardless of riskWhat a naive sales manager would do — "call your biggest accounts"

If the cluster-guided strategy can't beat "just call your biggest clients", the clustering isn't adding business value and should be re-tuned.

For each cluster, the validation report shows: number of partners, degrowth rate, average 90d revenue, average growth rate, average churn probability, average credit risk, and total estimated monthly loss. Degrowth Backtest Framework

monitoring_mixin.py → run_degrowth_backtest()

The degrowth alert says: "Partner X dropped more than 20% revenue in the last 90 days — flag as at risk." The backtest evaluates: was that flag actually predictive of further decline in the next 90 days? This turns the alerting rule from an assumption into a validated signal.

For each anchor month in the history window: prev_window = transactions from -180 to -90 days before anchor curr_window = transactions from -90 to 0 days before anchor next_window = transactions from 0 to +90 days after anchor drop_pct = (prev_revenue - curr_revenue) / prev_revenue × 100 pred_flag = drop_pct ≥ min_drop_pct (default: 20%) "The alert would have fired in this window" actual_flag = next_revenue < curr_revenue "Revenue actually continued to fall afterwards" Confusion matrix across all partner-window combinations: True Positive (TP): alert fired AND decline continued ← Good False Positive (FP): alert fired BUT revenue recovered ← Noise False Negative (FN): alert didn't fire BUT decline happened ← Missed precision = TP / (TP + FP) ← Of all alerts fired, what % were right? recall = TP / (TP + FN) ← Of all actual declines, what % did we catch?
Precision

Recall
Actual Declines Caught

ℹ️
How to interpret the backtest results
High precision, low recall: The alert is conservative — when it fires, it's usually right, but it misses many real declines. Reduce the threshold to catch more.
Low precision, high recall: The alert fires too often — it catches most real declines but generates a lot of false alarms. Increase the threshold.
Ideal range: Precision ≥ 0.65 and Recall ≥ 0.50 for a B2B degrowth signal is typically acceptable. The imbalance (precision vs recall) depends on business preference — false alerts are annoying, missed declines are costly.
09

Stock Exposure Score

monitoring_mixin.py → get_stock_details()

For products with slow-moving stock, the engine computes a stock exposure score from 0 to 100 that quantifies how urgently a product needs liquidation attention. The score feeds into the EOL module's clearance generator and the dead stock view.

Four components, each normalised to its contribution ceiling: age_component = min(max_age_days / 180, 1.0) × 55 ← Age (days) is the dominant signal percentile_component = age_percentile / 100 × 20 ← Relative age vs all products qty_component = min(total_stock_qty / 100, 1.0) × 15 ← Inventory volume pressure recency_component = min(demand_recency_days / 180, 1.0) × 10 ← Days since last buyer purchase stock_exposure_score = age_component + percentile_component + qty_component + recency_component Range: 0 – 100 Priority classification: score ≥ 80 → Critical "Immediate Action" score ≥ 60 → High "Plan Sales" score ≥ 35 → Medium "Monitor Weekly" score < 35 → Low "Monitor Monthly"
💡
Why age carries 55% of the weight Old unsold inventory is the primary risk in distribution. It ties up working capital, occupies warehouse space, and depreciates in value (especially for tech products). A product with moderate stock quantity but sitting for 200 days is a worse problem than a product with high stock quantity that was received 30 days ago. The 55% weight on age reflects this financial reality.
