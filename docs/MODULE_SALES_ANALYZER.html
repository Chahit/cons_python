02

Revenue Accuracy — Approved-Only Filter

Every revenue query applies a three-layer filter to ensure only genuine, approved transactions are counted. This was a critical fix from a previous session that eliminated 10–30% revenue inflation caused by cancelled or reversed transactions being included in totals.

Three mandatory join conditions on every revenue query: 1. is_approved check: WHERE LOWER(CAST(t.is_approved AS TEXT)) = 'true' ← Excludes cancelled/rejected DSR transactions 2. due_payment join: JOIN due_payment dp ON dp.dsr_id = t.id AND dp.is_active = TRUE AND dp.deleted_at IS NULL ← Excludes reversed/void payment records 3. Revenue column: net_amt (not gross_amt) ← net_amt already deducts returns/adjustments; gross_amt includes them Combined effect: 100% accurate revenue — no inflation from cancellations
🔴
Why this mattersWithout the due_payment join, reversed transactions still show is_approved=true but the payment record has is_active=false. Using gross_amt on top of that caused some partner revenues to appear 10–30% higher than actuals. This is now fixed across all Sales Analyzer queries.

Date Range Engine

_render_date_picker()

The date picker supports both quick preset chips and a custom calendar range. The default window is 5 years (to show all historical data), narrowable to any custom range the user defines. The active window is shown as a confirmation banner below the picker.

PresetDaysUse case
15 Days15Recent activity check — last fortnight's orders
1 Month30Monthly sales review
2 Months60Bi-monthly performance window
3 Months90Quarterly review (most common for partner health checks)
6 Months180Half-year trend analysis
1 Year365Annual partner performance comparison

The date range drives YoY comparison by shifting the window exactly span days back: yoy_end = start − 1d, yoy_start = yoy_end − span_days. This ensures the comparison period is always the same length as the current period, regardless of presets used.

5-Signal Next-Order Prediction

predict_next_order() in sales_analyzer_extras.py

The prediction engine uses the partner's complete transaction date history (no date filter) to compute a weighted next-order estimate. It uses 5 independent signals combined sequentially to account for recency bias, trend, and seasonality.

5-Signal Algorithm (applied sequentially): Signal 1 — Weighted-Recency Median Weight gaps linearly: oldest gap = 1×, most recent gap = 2× (10→20 units) w_median = median(weighted_pool) ← Gives more importance to recent ordering cadence than historical average Signal 2 — Trend Detection early_avg = avg(gaps[:n//3]) ← First third of gap history recent_avg = avg(gaps[-n//3:]) ← Last third of gap history IF recent_avg < early_avg × 0.85 → "accelerating" 📈 IF recent_avg > early_avg × 1.15 → "decelerating" 📉 ELSE → "stable" ➡️ Signal 3 — Trend-Adjusted Gap accelerating (n≥4): final_gap = w_median × 0.90 ← Shorten prediction decelerating (n≥4): final_gap = w_median × 1.10 ← Lengthen prediction stable: final_gap = w_median Signal 4 — Seasonal Adjustment same_month_gaps = gaps where dates[i].month == last_order.month IF len(same_month_gaps) ≥ 2: final_gap = final_gap × 0.6 + seasonal_median × 0.4 Signal 5 — Confidence Tier cv = std(gaps) / w_median cv < 0.25 AND n ≥ 6 → High cv < 0.50 AND n ≥ 3 → Medium else → Low
💡
Why weighted median instead of mean?A partner who ordered every 30 days for 2 years, then changed to every 60 days over the last 6 months, would show a mean of ~35 days — misleading. The weighted median gives the 6-month pattern 2× weight, correctly predicting ~60 days. Mean would dramatically underestimate the gap.
05

Repeat vs. New Product Classification

fetch_repeat_new() in sales_analyzer_extras.py

Each product in the selected date window is classified as Repeat or New at the product level (not order level). The definition deliberately avoids the "all-New" problem that occurs when the window starts at a partner's very first order.

Classification logic (SQL CASE): Repeat IF: → Product was bought by this partner in ANY transaction BEFORE current window (t.date < window_start) OR → Product was bought MORE THAN ONCE within the current window (COUNT(DISTINCT t.id) > 1 within window) New IF: → Product appears in current window for the FIRST TIME EVER (no prior purchases, bought only once in this window) Why two Repeat conditions? If window starts at first-ever order date: all products would be "New" even if the partner has ordered them hundreds of times. The in-window repeat condition catches this case.

Multi-Partner Comparison View

tab2 — Compare Partners

The Compare tab allows side-by-side analysis of up to 3 partners over the same date window. It produces KPI cards, an overlaid monthly revenue line chart, and a grouped category revenue bar chart.

KPI Cards
Revenue, Orders, Total Qty, Unique Products side-by-side. Uses fetch_multi_partner_kpis() — one query per partner looped, joined on company_name.
Monthly Revenue Chart
Overlaid line chart using Plotly Express. All partners' monthly revenue on the same axes. Color sequence: green → indigo → amber.
Category Revenue Comparison
Grouped bar chart: X = product category, Y = revenue, grouped by partner. Shows which partner dominates which category.
Deep-link from Kanban
st.session_state["sa_preselect_partner"] — Kanban pipeline writes this key; Sales Analyzer reads and pops it to auto-select the partner on load.

08

PDF Export & INR Formatting

export_pdf() · _fmt_inr()

The PDF export generates a landscape A4 dark-themed report using fpdf. Unicode characters (₹, →, etc.) are mapped to latin-1 equivalents via _pdf_safe() to avoid encoding errors with Helvetica.

PDF structure: Header bar (accent green, 24mm) → Partner name + city/state/date range KPI block (6 cards at 44mm each across the page width) Product table (top 25 products, truncated at 38 chars, alternating rows) Footer: generated date _pdf_safe() character map (critical for fpdf Helvetica compatibility): ₹ → Rs → → -> • → - — → - … → ... × → x

INR Number Formatting — _fmt_inr()

Indian number format used across all displays: val ≥ 1,00,00,000 → ₹{val/1_00_00_000:.2f} Cr (crores) val ≥ 1,00,000 → ₹{val/1_00_000:.2f} L (lakhs) val < 1,00,000 → ₹{val:,.0f} (exact with comma)
