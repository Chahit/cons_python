02

The Root Problem — Old Behaviour

The old High-Affinity Prospects tab used a single gate: if no association rules existed for the selected product, it immediately displayed an info banner and stopped — showing zero prospects. This happened silently for many products.

⚠️
Old code gate (now removed) if not trigger_prods: st.info("No co-purchase signal available...")
Products like "CON:DEZIRE White Wireless Mouse" — sold as a standalone SKU with thin basket overlap — always hit this gate. The tab showed nothing despite there being dozens of category-matched partners in the DB.
03

3-Tier Prospect Engine — Overview

recommendation_hub.py → lines 457–598

The new engine always attempts all three tiers in order of signal strength. Each tier is deduplicated against prior tiers — a partner only appears once, in their highest-signal tier.

Tier 1
🔵 Co-Purchase Signal
Partners who buy association-rule trigger products but never bought this one. Strongest signal — proven basket overlap. Source: MBA rules from get_associations().
Tier 2
🟡 Category Match
Partners who buy OTHER products from the same product group. Critical fallback — category-proven buyers already have procurement, storage, and customers for this product class.
Tier 3
⚪ Cluster Lookalike
Partners in the same behavioural clusters as existing buyers, scored by cosine similarity to the past-buyer centroid plus recency decay. Never shows generic top-revenue accounts.
💡
The tab never shows an empty state As long as the product has at least one buyer and belongs to a product group in master_group, the engine will always surface Tier 2 (category match) partners at minimum. Only a truly new SKU with no category peers and no buyers can produce zero results — and that case shows a specific, informative empty state instead of a generic "no data" message.
04

Tier 1 — Co-Purchase Signal

_fetch_product_prospects()

Tier 1 uses the existing Market Basket Analysis association rules engine. It collects trigger products from both directions of the rule (A→product and product→A) to maximise coverage, then queries the DB for all partners who buy those trigger products but have never bought the selected product.

Association rule directions (both collected): t_from_b = rules WHERE product_b == sel_product → product_a list ← sel_product is the consequent t_from_a = rules WHERE product_a == sel_product → product_b list ← sel_product is the antecedent trigger_prods = union(t_from_b + t_from_a) DB query: partners who buy trigger products but NOT the selected product tier1_df = _fetch_product_prospects(engine, sel_product, trigger_prods, existing_buyers) tier1_df["_signal"] = "🔵 Co-Purchase" tier1_df["_tier"] = 1
05

Tier 2 — Category Match

_fetch_product_meta() · _fetch_category_prospects()

When Tier 1 is sparse or empty, Tier 2 queries the DB directly for partners who buy any other product from the same product group but have never bought this specific SKU. These are category-proven buyers — the highest-conversion cold pool because the product pitch is entirely natural to their existing business.

Step 1: resolve the product's group_name group_name, category_name = _fetch_product_meta(engine, sel_product) → JOIN master_products → master_group → master_product_category Step 2: query category peers who haven't bought this SKU SQL: SELECT mp.company_name, mg.group_name, p.product_name AS trigger_product, SUM(tp.qty) AS trigger_qty, MAX(t.date) AS last_purchase_date FROM transactions_dsr t JOIN ... WHERE mg.group_name = %(group_name)s AND p.product_name != %(product_name)s GROUP BY company_name, product_name ORDER BY trigger_qty DESC Step 3: exclude existing_buyers + tier1 partners, dedup to best trigger per partner tier2_df["_signal"] = "🟡 Category Match" tier2_df["_tier"] = 2
ℹ️
Why is Category Match more valuable than a generic lookalike? A partner who buys CON:DEZIRE Black Wireless Mouse already: stocks accessories, has storage, knows the customer demand, has payment terms established. Pitching them the White variant is a conversation of seconds — "we have the white version, want to add it?" vs. introducing an entirely new product category requiring proof of demand.
06

Tier 3 — Cluster Lookalike (Geometric Scoring)

recommendation_hub.py → lines 513–582

Tier 3 uses the same cluster membership approach as the Inventory Liquidation engine but scored geometrically. Partners in the same behavioural clusters as existing buyers are scored by cosine similarity to the past-buyer centroid — ensuring that the lookalikes actually look like the buyers, not just share a cluster.

Tier 3 algorithm: 1. Find which clusters existing buyers belong to (top 2 clusters by count) 2. Build candidate pool: same clusters, exclude all buyers + tier1 + tier2 partners Geometric similarity (if df_partner_features available): feature_cols = [mix::rw::* cols] + [scale::* cols] buyer_centroid = mean(feature_vectors of all existing_buyers) geo_sim[i] = cosine_similarity(candidate_i_vector, buyer_centroid) Recency score: rec_score[i] = 1 / (1 + recency_days[i] / 30) Combined score: lookalike_score = 0.60 × geo_sim + 0.40 × rec_score sort DESC → top 15 candidates tier3_df["_signal"] = "⚪ Lookalike" tier3_df["_tier"] = 3
SignalWeightSource columnMeaning
Geometric Similarity60%mix::rw:: + scale:: via cosineHow close is this partner's behaviour profile to the centroid of actual past buyers?
Recency Score40%recency_daysDecay function — dormant partners rank last regardless of cluster similarity.

📌
"Why Recommended" column Every prospect row shows the specific trigger product — e.g. "CON:DEZIRE Black Wireless Mouse" for a category match, or "⚪ Cluster lookalike (VIP Stable)" for a tier 3 result. Sales reps immediately understand the pitch context without needing to run any additional queries.

AI Outreach Scripts (Partner → Product Tab)

_build_personalized_scripts()

For the Partner → Product recommendation tab, the system generates personalised WhatsApp and email outreach scripts for each recommended product. Scripts reference the partner's actual purchase history, cluster label, and health segment — avoiding generic discount language.

Script elementData sourcePurpose
Last product referencepartner_ctx["last_product"]Opens with a real product name the partner knows — builds trust immediately
Trigger productsAssociation rules → trigger_productsExplains WHY the recommendation is relevant ("partners who buy X also buy Y")
Cluster labelcluster_label from df_partner_featuresTailors tone — "VIP Stable" partners get a premium pitch; "At Risk" partners get urgency
Health segmentChurn/credit score → health bucketScripts are awareness-only for Critical partners; full pitch for Champion partners
