The Five Swimlanes — Lane Assignment Logic

kanban_pipeline.py → LANES config

Every partner lands in exactly one of five lanes. The lane is determined by the partner's health_segment value, which is computed by the ML engine's weighted health score formula. When the user switches the Time Period filter (Monthly / Quarterly / Yearly), the _recompute_segments_for_period() function re-classifies partners on the fly using scaled revenue thresholds and two Champion qualification paths.

Lane Header Metrics

Each lane header shows three numbers — not just a partner count:

lane_count = number of partners currently in this lane lane_value = sum of recent_90_revenue for all partners in this lane lane_at_risk = sum of expected_revenue_at_risk_90d for all partners in this lane The "At Risk Revenue" row only appears if > ₹0 — so Champion and Healthy lanes typically show only the total pipeline value, not a red risk number.

This lane-level summary immediately answers the most important sales management question without even opening a single card: "How much revenue is sitting in risky hands?"

Card Performance Cap

Each lane shows a maximum of 50 cards. If a lane has more partners, a caption at the bottom says "Showing top 50 of N — use filters to narrow down." The cap is for browser performance: rendering 200+ expandable cards with embedded HTML causes lag in Streamlit. 50 per lane (× 4 lanes = 200 total) is the empirically safe limit on typical laptop hardware.

Churn Colour Coding — The Visual Risk Signal

kanban_pipeline.py → _churn_border_color()

The churn probability value is shown as both a percentage and a colour — so a manager scanning cards doesn't need to read numbers to spot high-risk partners. The colour is also applied to the card's left border, making the risk visible even before the card is expanded.

Churn probability → CSS border/text colour mapping: churn ≥ 0.70 → #ef4444 (red) — Critical risk: call immediately churn ≥ 0.50 → #f59e0b (amber) — High risk: schedule this week churn ≥ 0.30 → #3b82f6 (blue) — Watch list: next regular visit churn < 0.30 → #22c55e (green) — Stable: no action needed
< 30% — Stable
30–50% — Watch
50–70% — High Risk
≥ 70% — Critical
🎨
Why four colours and not a simple red/green? In a typical B2B distribution network, 30–50% of partners will have a churn probability between 30–65% at any given time — they are neither obviously fine nor obviously at risk. A red/green binary would either mark everything as red (panic) or everything as green (complacency). The four-colour scale lets managers triage: respond to red today, plan for amber this week, check blue next visit, ignore green.

Revenue Trend Arrows — Reading Growth vs Decline

kanban_pipeline.py → _trend_arrow() + _last_seen()

The revenue trend arrow converts revenue_drop_pct into a visual signal. Note that "drop percent" is used — so a positive value means the partner is declining, a negative value means they're actually growing.

revenue_drop_pct value → Arrow display logic: drop ≥ 30 → ▼ 30% (red) "Sharp decline" drop ≥ 10 → ▼ 15% (amber) "Moderate decline" drop ≤ −10 → ▲ 12% (green) "Growing" (negative drop = growth) −10 < drop < 10 → → 4% (grey) "Stable / minimal change" Example: partner grew 28% → revenue_drop_pct = -28 → shown as ▲ 28% (green) Example: partner fell 45% → revenue_drop_pct = +45 → shown as ▼ 45% (red)

Last Seen (Recency) Display

Raw recency_days is converted to a human-readable string — no one reads "recency_days = 47" and feels urgency, but "2mo ago" is immediately understood as a problem:

recency_days = 0 → "Today" recency_days = 1 → "Yesterday" recency_days ≤ 7 → "{d}d ago" e.g. "5d ago" recency_days ≤ 30 → "{d÷7}w ago" e.g. "2w ago" recency_days ≤ 365 → "{d÷30}mo ago" e.g. "3mo ago" recency_days > 365 → "{d÷365}y ago" e.g. "1y ago"

Revenue at Risk — Calculation and Fallback

kanban_pipeline.py → _build_kanban_df() fallback logic

This is the most important metric on the board from a business standpoint. Revenue at risk is not just "this partner's revenue" — it is probability-weighted revenue loss. A partner with ₹10L in revenue and 80% churn probability carries ₹8L at risk. That is the number that should drive management attention.

Fallback Formula — Inline Computation

In fast mode (FAST_MODE=true), full churn scoring is skipped to save load time. This means expected_revenue_at_risk_90d arrives as zero for most partners — even though churn_probability is computed. The Kanban detects this inconsistency and fills it in:

Detect inconsistency: rev_at_risk == 0 AND churn_probability > 0 fallback_mask = (expected_revenue_at_risk_90d == 0) AND (churn_probability > 0) Apply fallback formula (same as churn scoring engine): expected_revenue_at_risk_90d = churn_probability × recent_90_revenue Examples: Churn = 80%, Revenue = ₹5L → At Risk = ₹4.0L Churn = 45%, Revenue = ₹2L → At Risk = ₹0.9L Churn = 10%, Revenue = ₹8L → At Risk = ₹0.8L
✅
Why is this called probability-weighted loss — not worst-case? If we showed worst-case (i.e., the full revenue of every at-risk partner), the total "revenue at risk" metric would always look catastrophic. A partner with 5% churn and ₹20L revenue would contribute ₹20L to the risk figure, which is misleading — their probability-adjusted risk is only ₹1L. By using churn_probability × revenue, we show the expected monetary loss if the model's predictions are correct on average across the whole portfolio.

Indian Number Formatting — Why We Built a Custom Function

kanban_pipeline.py → _fmt_inr()

Python's built-in number formatting produces "₹1,234,567" — which is the international format. Indian business uses a different system: ₹12.3L (lakhs) and ₹1.2Cr (crore). Every revenue figure on the dashboard uses this custom formatter.

_fmt_inr() logic: value ≥ 1,00,00,000 → f"₹{value / 1_00_00_000:.1f}Cr" (crores) value ≥ 1,00,000 → f"₹{value / 1_00_000:.1f}L" (lakhs) value ≥ 1,000 → f"₹{value / 1_000:.0f}K" (thousands) otherwise → f"₹{value:.0f}" (absolute) Applied consistently across: card headers, lane total, board summary, at-risk values

Filters and Sort Options

kanban_pipeline.py → sidebar filters + sort

Five filter/sort controls are available in the sidebar and on the main page. They all apply before the board renders — so if you filter to "Maharashtra + High Credit Risk" and sort by churn, every lane shows only the matching partners in churn order.

ControlTypeWhat it doesDefault
Time Period Dropdown Monthly / Quarterly / Yearly. Scales revenue figures and re-runs segment classification via _recompute_segments_for_period(). Quarterly = model default (no recalc). Quarterly
Filter by State Multi-select Shows partners from selected states only. Dynamically populated from data. Empty = all states. All states
Filter by Credit Risk Dropdown Restrict to Low / Medium / High / Critical credit bands. "All" shows everyone. All
Sort Cards by Dropdown Revenue (High→Low) / Churn Risk (High→Low) / Revenue Drop (High→Low) / Name (A→Z) Revenue (High→Low)
Min 90d Revenue Number input Hides partners below a revenue floor. e.g., ₹1,00,000 removes microdistributors from view. ₹0 (no filter)
Search by Name Text search Case-insensitive substring match on company_name. Filters all lanes simultaneously. None
ℹ️
All filters are additive (AND logic) If you pick "Maharashtra" AND "High credit risk" AND "min revenue ₹50,000", you see only partners that satisfy all three conditions simultaneously. This is intentional — a sales manager doing a focused session (say, collections in Maharashtra) needs to see exactly that slice, not everything that matches any one condition.

Performance Design & Improvements

Caching Strategy

@st.cache_data(ttl=300, show_spinner=False) def _build_kanban_df(_pf_df): ... The heavy data slicing (selecting only needed columns, filling defaults, computing fallback revenue-at-risk) runs once and is cached for 5 minutes. All subsequent filter/sort operations run on the cached slice — not the full matrix. Result: first load takes ~1–2 seconds. Subsequent filter changes: <100ms.

Skeleton Loader

While clustering and partner features are being loaded, the page shows a placeholder UI (skeleton loader) instead of a blank screen. This makes the dashboard feel responsive even when the ML engine is running — identical to how modern banking apps show empty card outlines while data loads.
