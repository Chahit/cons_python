The AI Chatbot is the only module in the system that talks back. Rather than reading tables and charts, a manager types a plain English question — "which partners are about to churn?", "who are the top sales reps this month?", "what products should I push to partner XYZ?" — and the system answers in 2–3 seconds with real numbers from the live database. The chatbot doesn't guess. It only ever answers using data from the last run of the dashboard, and is explicitly instructed not to invent figures. It's a business intelligence layer that removes the need to navigate seven tabs to answer one question.

01

Three-Layer Architecture

The chatbot is not a simple "hook GPT to a database" implementation. It has three distinct layers, each with a specific responsibility:

Layer 1
Data Layer
Module Pre-Loader + In-Memory Cache All 7 modules are pre-loaded into Python memory on the first chat message. After that, all data is served from RAM — no database queries during the chat itself. This makes responses fast (2–3 seconds vs 10–20 seconds if we queried DB on every question).
Layer 2
Context Layer
Keyword Intent Router + Context Builder Each question is scanned for keywords. Matching keywords activate specific data extracts. Only the relevant data is assembled into a structured text block (the "context") that is sent to GPT. This prevents sending 50,000 tokens of irrelevant data into every API call.
Layer 3
Reasoning Layer
GPT-4o — Language + Logic The LLM receives the system prompt (persona + rules), the context (live data), the conversation history (last 8 turns), and the user's question. It interprets, synthesises, and formats a response. GPT never accesses the database — it only sees what the context builder has extracted and formatted.
🎯
Why this architecture specifically? The alternative (RAG — retrieval-augmented generation) would involve embedding the database content into a vector store and retrieving relevant chunks on demand. RAG is powerful for general document Q&A, but in this case, the data is already structured Python dataframes — not unstructured text. Keyword routing on pre-loaded in-memory data is faster, simpler, and equally accurate for structured business intelligence. The RAG tradeoff is covered in detail in Section 09.
02

One-Time Module Pre-Loader

chatbot_mixin.py → _ensure_all_modules()

The first time anyone sends a message in the chatbot, a single pre-load function runs. It calls every module's ensure_*() method sequentially. Each method is idempotent — if the data is already loaded, it returns immediately. After the pre-load, the _chatbot_all_loaded flag is set to True, preventing this from running again during the same session.

Pre-load sequence (runs ONCE on first chat message): modules_to_load = [ "ensure_core_loaded", ← partner features, health scores, revenue data "ensure_churn_forecast", ← churn probabilities and revenue at risk "ensure_credit_risk", ← credit scores, overdue ratios, outstanding amounts "ensure_clustering", ← cluster labels, cluster types, silhouette scores "ensure_associations", ← MBA rules: lift, confidence, times_bought_together "ensure_product_lifecycle", ← velocity scores, EOL predictions, cannibalization "ensure_sales_rep_data", ← leaderboard, ROI, tours, expenses, issues ] Each call is wrapped in try/except — one failing module does not crash the chatbot. The chatbot gracefully replies with partial data if some modules failed. After first load, _chatbot_all_loaded = True prevents re-running. Subsequent messages: all data is served from memory in milliseconds.
💡
Why pre-load everything at once instead of loading on demand? If we loaded modules on demand (only when the relevant keyword appears), the first question about "churn" would take 10 seconds to load. The second question about "clusters" would also take 10 seconds. By paying the one-time cost upfront when the user sends their first message, every subsequent question is instant. This is the right tradeoff for a conversational interface where response time under 3 seconds is critical to perceived quality.

04

Partner Name Fuzzy Finder

chatbot_mixin.py → _find_partners()

When a user mentions a partner name in their question, the system needs to identify whether a known partner is referenced — even if the name is slightly different from how it's stored in the database (abbreviated, mis-spelled, partial name). The fuzzy finder works in two passes:

Pass 1 — Exact substring match (fast path): exact = [p for p in all_partner_names if p.lower() in question.lower()] "raj paints" in "what's the status of raj paints" → match Returns up to 3 exact matches. Used when the name appears verbatim. Pass 2 — Word overlap scoring (fuzzy fallback): words = [w for w in question.split() if len(w) >= 4] ← ignore short words For each candidate partner name: score = count(words in question that appear inside partner name) e.g. question: "what about raju enterprises sales" candidate: "Raju Enterprises Ltd" words: ["what", "about", "raju", "enterprises", "sales"] score = 2 (raju + enterprises match in partner name) Sort all candidates by score descending. Return top 3 with score > 0. Returns [] if no partner name appears to match — prevents false injections.
⚠️
Why not use a proper fuzzy matching library (e.g., fuzzywuzzy / rapidfuzz)? Two reasons: (1) it's an optional dependency that might not be installed, and (2) character-level fuzzy matching generates too many false positives on short partner names in a B2B context — "Raj" would match "Raj Paints", "Rajesh Hardware", and "H.K. Raj & Sons" all with similar scores. The word-overlap approach is more precise for B2B partner name disambiguation, which typically involves multi-word company names with at least one distinctive word.
05

Context Builder — What Goes to GPT

chatbot_mixin.py → _build_chat_context() + _get_full_partner_context()

The context is a structured text block, assembled fresh for every question, that contains only the data relevant to this specific question. It's a carefully curated subset of the in-memory data that fits within the GPT context window efficiently.

Always Included
Business Snapshot: Total partners, health breakdown, high-churn count (>65%), total forecasted revenue (all partners, next 30d), total all-time revenue, last 12 months revenue. This ensures GPT always has the big picture even for narrow questions.
Partner Profile (if name found)
28 fields per partner: health status, health score, cluster, lifetime revenue, recent 90d revenue, revenue drop %, churn probability, churn band, revenue at risk, forecast next 30d, credit risk, overdue ratio, outstanding amount, top cross-sell pitch, pitch confidence, last 4 months revenue history, top product groups.
Churn Section
Top 25 partners by churn_probability, total estimated monthly revenue at risk across all partners, top 10 by expected_revenue_at_risk_90d.
Revenue Section
Monthly total revenue — last 18 months. Top 10 partners by revenue in the last 3 months. Top 15 partners by forecast_next_30d. Total projected revenue across all partners.
Product / MBA Section
Top 15 bundle rules (product A → product B with lift, confidence, times_bought_together). Product groups ranked by total revenue. Top buyers of any specifically mentioned product group.
Fallback Context
If only ≤ 1 section matches and no partner name was found, the system injects a "Top 10 Partners by Revenue" block as a fallback baseline. This prevents GPT from receiving an empty context and hallucinating data.

Full Partner Context — 28 Fields

When a partner name is identified, all 28 available data fields are extracted from both the get_partner_intelligence() report and the df_partner_features matrix. Fields are formatted based on their type:

Format types applied to each field: money → "Rs 12,34,567" ← Indian comma format pct100 → "34.5%" ← score × 100 pct → "−12.3%" ← already in percentage scale int → "47" ← integer count score → "0.743" ← 3 decimal places float2 → "2.34" ← lift, confidence values text → as-is string ← health band, cluster label, etc. Null safety: NaN, "None", "N/A", "" → field skipped entirely This prevents "Churn Probability: nan%" being sent to GPT, which it might try to interpret as a real value. Merge priority (report wins over df_partner_features on conflicts): merged = {**df_partner_features_row, **get_partner_intelligence_report} ↑ lower priority ↑ higher priority (more computed fields)

System Prompt — Rules & Persona

chatbot_mixin.py → chat_with_ai() → system_prompt

The system prompt is sent at the beginning of every API call. It establishes who the AI is, what data it has access to, and the strict rules it must follow. This is the most important engineering decision in the chatbot — it prevents hallucination and enforces business-appropriate output format.

PERSONA: "You are a senior business intelligence assistant for CONSISTENT — a B2B distributor of lubricants, paints, greases, construction chemicals, and industrial products across India." AVAILABLE MODULES DECLARED: Module 1: Partner 360, Module 2: Market Basket, Module 3: Clusters, Module 4: Inventory, Module 5: Product Lifecycle, Module 6: Recommendations, Module 7: Sales Rep, Revenue Pipeline Tracker STRICT RULES (7 rules, all enforced): Rule 1: ONLY use data provided below. NEVER invent names, numbers, or figures. Rule 2: "PROJECTED REVENUE (Next 30d)" = the partner's forecast. Quote it exactly. Rule 3: Churn probability is a decimal — multiply by 100 for percentage display. Rule 4: Format currency as Rs X,XX,XXX (Indian number format — NOT $) Rule 5: If a partner not found: "Couldn't find [name] — check exact spelling." Rule 6: Answer directly. No "I can see in the data..." conversational filler. Rule 7: Keep responses under 300 words. Lead with the most important number. THEN: The full context block (relevant sections from _build_chat_context())
💡
Why is Rule 1 (no hallucination) the most critical? LLMs are probabilistic — they will fill in plausible-sounding data if none is provided. In a B2B sales context, if GPT invents "Partner XYZ has 78% churn risk" and it's actually 20%, a sales manager might call that partner unnecessarily. Rule 1, combined with the data injection in the context block, grounds the model in reality. The fallback context (Section 05) ensures GPT always has some data to work with rather than nothing — reducing the incentive to hallucinate.
07

Conversation History Management

The chatbot maintains multi-turn conversation context. When the user asks a follow-up question ("and what about their credit risk?"), GPT can refer to the previous exchange to understand what "their" refers to. This is managed via the history parameter passed to each API call.

Message list structure sent to GPT on each call: messages = [ {"role": "system", "content": system_prompt + context}, ← always first {"role": "user", "content": message_1}, ← prior turns {"role": "assistant", "content": response_1}, ← ↑ ... ↑ last 8 turns {"role": "user", "content": current_question}, ← current question ] History window: last 8 turns (4 exchanges) — history[-8:] Why 8 turns and not unlimited? GPT-4o has a large context window (128K tokens), but each turn adds tokens. The system prompt + context can be 3,000–8,000 tokens depending on how many modules are activated. Adding 10+ history turns would push toward the model's effective reasoning limit where early turns receive less attention. 8 turns ≈ 4 back-and-forth exchanges — sufficient for any natural business Q&A. Note: the CONTEXT is regenerated fresh on every call based on the NEW question. History is the conversation; context is fresh data.

GPT-4o Parameters Explained

Every parameter in the GPT-4o API call is chosen deliberately. This is not a default configuration.

model
gpt-4o (configurable)
GPT-4o has the best reasoning-to-cost ratio for structured data interpretation. Configurable via OPENAI_MODEL in .env — you can swap to gpt-4o-mini for lower cost or gpt-4-turbo for higher capacity.
temperature
0.05
Near-zero. Factual business intelligence answers must be deterministic — the same question about churn risk should always get the same answer if the data hasn't changed. Temperature 0.05 provides minimal variation while avoiding degenerate outputs from pure 0.
max_tokens
600
Maps to Rule 7 (under 300 words). 600 tokens ≈ 400–450 words — slightly more than the target, ensuring the model isn't cut off mid-sentence if it slightly exceeds the guideline. Prevents runaway responses that fill the screen.
history window
history[-8:]
Last 8 messages (4 exchanges). Balances multi-turn coherence with token budget. Fresh context is injected on every call — old context from prior questions doesn't accumulate in history.

Why Not RAG? — Architecture Comparison

Retrieval-Augmented Generation (RAG) is the industry-standard approach for LLM-powered Q&A over large data corpora — systems like LlamaIndex, LangChain, and Pinecone are built around it. Here's why the keyword-routing approach was chosen instead, and what RAG would offer if implemented.

DimensionCurrent approach (Keyword Router)RAG (Embedding + Vector Search)
Setup complexity Zero — pure Python, no additional infrastructure Requires a vector database (Pinecone / Chroma / Weaviate), embedding calls, chunking strategy
Latency ~2–3 seconds (context build + API call) ~3–5 seconds (embed query + vector search + API call)
Data freshness Always live — in-memory data from latest run Stale if vector store not refreshed with every DB update
Answer precision for structured queries High — exact numbers, exact partner names from dataframes Depends on chunk quality — numbers can be retrieved from wrong context chunks
Scalability to large text corpora Poor — keyword routing doesn't scale to unstructured text (e.g. call notes, emails) Excellent — RAG is built for this
Handling ambiguous questions Relies on GPT to interpret ambiguous intent after seeing multi-section context Better — semantic similarity retrieval handles paraphrasing and implicit intent
Best for Structured tabular data, known question types, fast deployment Document Q&A, call notes, emails, policy documents, mixed content types
ℹ️
The verdict: right tool for the right job For the current use case — structured partner data, product data, rep metrics, all in Python dataframes — the keyword router wins on simplicity and data freshness. The moment the system needs to answer questions about unstructured content (e.g. "what did the sales rep note after visiting Raj Paints last month?"), RAG becomes necessary. That's the natural next evolution.

Limitations & Improvements

#LimitationCurrent stateImprovement path
1 Keyword routing is brittle If a user asks "What companies are at the highest risk?" the word "risk" matches the churn section, but "at highest risk" might semantically refer to credit risk — the current system returns both sections but can't interpret which one the question is primarily about. Add a lightweight intent classification step using GPT (a very fast, cheap gpt-4o-mini call with a strict JSON output schema: "intent": "churn|credit|revenue|...") before building the context. This adds ~200ms but dramatically improves routing precision.
2 No "ask for clarification" capability If the question is ambiguous and no keywords match, the chatbot returns the fallback "Top 10 Partners" context, which is often irrelevant to the actual question. Add a clarification condition: if fewer than 2 context sections match and no partner was found, return a suggested question list rather than a generic top-10 list.
3 No audit trail of chatbot queries Questions asked and answers given are not stored. If management asks later what recommendations were made to which partner, there's no log. Log every (question, context_summary, response, timestamp, user_id) tuple to a DB table. This also enables future fine-tuning of the chatbot on actual business queries.
4 Partial RAG for notes and emails Call notes, visit reports, and email summaries are not in the system — questions like "what was said to partner X last visit?" can't be answered. Add a second context path: a vector store (Chroma embedded in local storage) for unstructured content. The keyword router sends structured questions to in-memory dataframes and text questions to the vector store.
5 Single LLM dependency The entire chatbot is coupled to OpenAI GPT-4o. If the API is down, the chatbot is completely unavailable. Add a fallback LLM (e.g. Gemini Flash via Google API). If OpenAI returns an error, retry once with the fallback. Most responses would still work — both models handle structured data well.
