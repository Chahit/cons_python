Why Clustering Matters — And What Breaks Without It

Clustering is the foundation of the entire intelligence engine. Every peer comparison, every cross-sell gap, every "similar partner" badge in the recommendation hub depends on having meaningful, stable, business-relevant groups.

Without clustering, a small regional distributor buying ₹3L/month in feature phones would be benchmarked against metro key accounts buying ₹50L/month across 12 categories. The gaps would be enormous, irrelevant, and would destroy trust.

The 7 Feature Groups

_build_cluster_features()

The engine builds features from 7 distinct behavioural signals, grouped by namespace prefix and combined into a single feature matrix before clustering.

GroupPrefixFeaturesWhat it captures
Category Mixmix::rw::Recency-weighted mix across 90d/180d/365dWhat type of buyer — product category shares. Primary clustering signal.
Scalescale::log_total_spend, active_groups, portfolio_concentration, active_group_momentumAccount size, breadth, and momentum
RFMrfm::log_recency, log_frequency, log_monetaryClassic Recency-Frequency-Monetary
Velocityvel::log_mean_gap, gap_cvBuying rhythm regularity
Diversitydiv::entropy, entropy_norm (Shannon)How spread out is spend across categories
Seasonalityseason::monthly_cv, seasonal_amplitude, season_q1–q4 (one-hot)Festival-driven vs steady buyers
Networknet::net_mean_peer_sim, net_max_peer_sim, net_centralityPortfolio similarity to peers (cosine)

Recency-Weighted Category Mix (The Key Formula)

For each time window, compute category share of spend: mix_90d = category spend in last 90 days ÷ total spend in last 90 days mix_180d = category spend in last 180 days ÷ total spend in last 180 days mix_365d = category spend in last 365 days ÷ total spend in last 365 days Recency-weighted combination (50/30/20 weighting): mix::rw::{category} = 0.5 × mix_90d + 0.3 × mix_180d + 0.2 × mix_365d Result: recent behaviour changes are reflected immediately without manual re-tagging.

Scale: Portfolio Concentration (HHI)

portfolio_concentration = Σ(category_share²) Range: 1/N (diversified) to 1.0 (all spend in one category) active_group_momentum = active_groups_90d − active_groups_365d Positive = expanding portfolio, Negative = contracting
03

Shannon Entropy — Measuring Category Diversity

_compute_category_entropy()
H = −Σ [ p_i × log₂(p_i) ] where p_i = share of category i H_norm = H ÷ log₂(N) where N = number of categories (0 to 1 scale) Examples: 100% mobiles only → H = 0 (zero diversity) Split evenly across 4 cats → H ≈ 2.0 (H_norm = 1.0) 70% mobiles + 20% + 10% → H ≈ 1.16 (H_norm ≈ 0.73)
🔍
Why not just count categories? A partner spending ₹9.5L in mobiles and ₹0.5L in 3 others looks diverse by count (4 categories) but is highly concentrated. Entropy captures this nuance — the more unequal the distribution, the lower the score.
04

Network Co-Purchase Similarity

_compute_network_features()
1. Build spend vector per partner → normalise to unit length 2. Compute pairwise cosine similarity (self-similarity excluded) cosine_sim(A, B) = (A · B) ÷ (|A| × |B|) Derived features (top-K = 5 neighbours): net_mean_peer_sim = avg cosine sim to top-5 most similar partners net_max_peer_sim = highest cosine sim to any partner net_centrality = mean cosine sim to ALL partners (hub score)
💡
Business meaning of centrality High centrality = "typical" distributor, safe for standard recommendations. Low centrality = unique buying pattern, needs customised attention. The algorithm often assigns low-centrality partners to outlier buckets.

Feature Preprocessing & Quality Guardrails

_feature_quality_guardrails()

Before clustering, the combined feature matrix passes through quality guardrails and preprocessing:

1
Null Handling
All NaN values filled with 0.0. Max null ratio per column is logged for monitoring.
2
Zero-Variance Pruning
Columns with standard deviation < 1e-9 are dropped. Prevents constant features from polluting distance calculations. If ALL columns pruned, a dummy_feature column is injected.
3
Mix Sum Consistency
For mix::rw:: features, validates that each row sums to ≈1.0. Reports mean absolute error (mix_sum_mae) for auditing.
4
RobustScaler
Scales using median & IQR instead of mean/std — a VIP spending 50× the average won't compress everyone else to near-zero.
⚙️
v3.0 change: No more UMAP The previous PCA → UMAP pipeline has been replaced. Features are now sent directly to the ensemble after RobustScaler. This eliminates the umap-learn dependency and removes a stochastic step that caused instability in smaller datasets.

Ensemble Consensus Clustering

_ensemble_segment()

The core clustering engine runs multiple algorithms at the same forced K, builds a consensus co-association matrix, and derives final labels via hierarchical clustering on the consensus. This is the biggest change from v2.

Algorithm Selection by Tier

TierAlgorithms RunConsensus Method
VIPKMeans (n_init=15) + GMM (full/diag covariance) + Spectral (k-NN affinity)Forced-K consensus → Agglomerative (average linkage) on distance = 1−C
GrowthKMeans (n_init=15) + GMM + HDBSCAN (leaf selection, ≤25% outlier gate)Forced-K consensus (excludes HDBSCAN if outlier-heavy) → same hierarchical cut

Business-Rule K Selection

Target: ~12 partners per cluster (actionable for sales teams) business_k = ceil(n_partners ÷ target_cluster_size) best_k = clamp(business_k, min=4, max=15) All algorithms run at this SAME K for label consistency. No silhouette-based K search — K is fixed by business rule.

Consensus Matrix Construction

For each pair of partners (i, j) across M algorithm runs: C[i,j] = fraction of runs where i and j share the same cluster (outlier labels −1 excluded from co-assignment counts) Range: 0.0 (never co-clustered) → 1.0 (always co-clustered) Final labels: distance = 1.0 − C[i,j] → AgglomerativeClustering(metric='precomputed', linkage='average') → Cut at target K → If Agglomerative fails: fallback to KMeans on consensus matrix
ℹ️
Why force all algorithms to the same K? In v2, each algorithm chose its own K, leading to consensus matrices where different algorithms disagreed on granularity (e.g., HDBSCAN found 3 clusters while GMM found 8). The consensus was noisy. Forcing a common K makes the co-association signal much stronger and more interpretable.

Growth Segment: Outlier Reassignment

For each partner with label = −1 (outlier): 1. Compute distance to every non-outlier cluster centroid 2. Find nearest cluster Reassign IF distance ≤ cutoff × multiplier cutoff = 70th percentile of intra-cluster distances base_mult = 0.9 (conservative for standard partners) high_mult = 1.1 (more permissive for high-value partners) high_value = partners above 80th percentile of log_total_spend Partners still outside range → remain "Uncategorized"

VIP vs Growth — Pareto Revenue Tiering

run_clustering()

Before clustering, partners are split into VIP and Growth tiers using a Pareto revenue split — not a fixed percentile cutoff.

Step 1: Sort partners by total spend (descending) Step 2: Compute cumulative spend share vip_revenue_share = 0.60 (top 60% of cumulative revenue) vip_max_share = 0.35 (at most 35% of partners can be VIP) vip_min_count = max(3, 8% of partners) Partners whose cumulative contribution ≤ 60% → VIP tier Remaining partners → Growth tier Bounds enforced: min 3 VIP, max 35% of base Safety: if Growth has < 2 partners, force 70/30 split
🔍
Why Pareto instead of fixed 80th percentile? A fixed percentile treats a top-20 partner base the same whether it contains 50 or 500 partners. Pareto adapts: in a concentrated network (few large accounts), the VIP tier is small and focused; in a distributed network, it's broader. This matches real B2B distribution dynamics.

Business Archetype Labelling — LLM + Heuristic

_auto_label_clusters()

Every cluster gets a human-readable business label. The system tries Gemini LLM first, then falls back to a rule-based heuristic.

Step 1: Build Centroid Profiles

For each cluster, a text profile is generated from: spend level (High/Medium/Low), portfolio breadth (diversified/moderate/focused), HHI concentration, and top 5 categories with spend share %.

Step 2: Gemini LLM Labelling (Primary)

If GEMINI_API_KEY is present: Send all cluster profiles to Gemini with structured prompt Request: 2-4 word title-cased labels per cluster Validate: non-empty string, 3-80 chars, no JSON artifacts Accept if ≥ 50% of clusters get valid labels

Step 3: Heuristic Fallback

Rule-based label construction from profile keywords: Spend level mapping: High-spend → "High-Value" | Medium-spend → "Mid-Tier" | Low-spend → "Emerging" Breadth mapping: diversified → "Diversified" | moderate → "Balanced" | focused → "Specialist" Concentration mapping: highly concentrated → "Category-Focused" | well-distributed → "Multi-Category" Final label format: {Tier} — {SpendLevel} · {Breadth|Concentration} Example: "VIP — High-Value · Diversified" Deduplication: if label collision, append #cluster_num

Cluster Quality Validation

_compute_quality_scores() · _estimate_stability()
Silhouette
> 0.4
Measures cluster separation. Range: −1 to +1. Above 0.4 is good for real-world B2B data.
Calinski-Harabasz
Higher = Better
Ratio of between-cluster to within-cluster variance. Compact + well-separated simultaneously.
Bootstrap ARI
≈ 1.0
5 runs on 80% subsets using KMeans resampling. ARI=1 = identical, <0.3 = unstable.

Stability Estimation (Bootstrap ARI)

For each of 5 bootstrap runs: sample = 80% of partners (random, no replacement, min 10) Re-cluster sample using same algorithm (KMeans for consensus) Compute ARI(base_labels[sample], new_labels) stability_score = mean(ARI across 5 runs) ARI = 1.0 → identical every time ARI = 0.0 → no better than random ARI < 0.3 → unstable, flag for review

Quality Gate & Auto-Fallback

_cluster_quality_gate() · _persist_cluster_run()

Every clustering run passes through a quality gate. If the gate fails, the system automatically rolls back to the last approved assignments from the database.

Quality gate checks (all must pass): 1. cluster_count ≥ CLUSTER_MIN_COUNT (default: 2) 2. outlier_ratio within [CLUSTER_OUTLIER_MIN, CLUSTER_OUTLIER_MAX] 3. VIP stability_ari ≥ CLUSTER_MIN_STABILITY_ARI (default: 0.0) 4. Growth stability_ari ≥ CLUSTER_MIN_STABILITY_ARI If gate PASSES: → Save run to cluster_model_runs (approved=true) → Save assignments to cluster_assignments table If gate FAILS: → Save run to cluster_model_runs (approved=false, reject_reason) → Load last approved assignments from DB and overwrite current matrix → If no historical approved run exists → use current (degraded) result
🛡️
Why auto-fallback matters Without fallback, a single bad data load (e.g., missing transactions) could cause all partners to collapse into 1 cluster, wiping out all peer comparisons and triggering wrong recommendations for every sales rep simultaneously. The quality gate prevents this cascade.

Temporal Stability Tracking

_match_clusters_hungarian() · _track_temporal_transitions()

On every clustering run, the system compares current assignments against the last approved run using the Hungarian algorithm to detect promotions, demotions, and cluster dissolutions.

Hungarian Algorithm for Label Matching

Problem: cluster labels are arbitrary (0, 1, 2...) across runs. "Cluster 0" today may correspond to "Cluster 3" yesterday. Solution: match by maximum partner overlap cost[new_i, old_j] = −overlap(partners in new_i ∩ partners in old_j) → Hungarian algorithm minimises total cost in O(n³) → Produces mapping: {new_label → prev_label} Only maps pairs with actual overlap > 0.

Transition Detection

For each partner present in both runs: Stable: same cluster (or matched cluster) Promoted: Growth → VIP tier change Demoted: VIP → Growth tier change Reassigned: moved to different cluster within same tier Output: stability_score = stable_count / total_common_partners Also reports: new_clusters, dissolved_clusters, top 50 transition details

Feature Drift Detection

_compute_feature_drift()
mean_abs_z_shift = mean(| (current_mean − baseline_mean) ÷ baseline_std |) Interpretation: z < 0.5 → normal variation z 0.5–1.0 → moderate shift, investigate pipeline z > 1.0 → significant change (new partners, seasonal shift, or data issue) Baseline updates on every run → adapts to real business trends. A sudden spike (0.3 → 2.1) is the actionable signal.

Business Validation Benchmarking

run_cluster_business_validation()

After every clustering run, the system automatically benchmarks cluster-guided targeting against two baselines to prove clustering adds business value.

Three targeting strategies compared (top 20% selection): 1. Cluster-Guided: rank clusters by avg churn + total loss, then rank partners within clusters by: 0.7 × churn_probability + 0.3 × normalised_loss Select top 20% of partners 2. Top-Revenue Baseline: select top 20% by recent_90_revenue 3. Random Baseline: random 20% sample Each strategy evaluated on: - hit_rate_degrowth_pct: % of selected partners with degrowth flag - captured_est_monthly_loss: total estimated monthly loss in selection - avg_churn_probability: average churn risk in selection
📊
Expected result Cluster-guided targeting should capture significantly more at-risk revenue than top-revenue or random baselines. If it doesn't, the clustering may not be capturing meaningful risk segmentation — a signal to revisit feature engineering.

v3.1 Bug Fixes

clustering.py

Bug Fix 1 — Series Ambiguity Crash

🐛
ValueError: truth value of a Series is ambiguous pf_cluster.loc[partner] returns a single-row DataFrame (not a Series) when the index has any duplicate entries. Calling .get(key, 0) or 0 on a DataFrame triggers the pandas bool check, which raises this error.
Root cause: pf_row = pf_cluster.loc[partner] # Could be Series OR DataFrame float(pf_row.get("revenue", 0) or 0) # Fails if pf_row is DataFrame Fix applied: _raw = pf_cluster.loc[partner] if isinstance(_raw, pd.DataFrame): pf_row = _raw.iloc[0].to_dict() # Normalise to plain dict else: pf_row = _raw.to_dict() def _scalar(key, default=0): v = pf_row.get(key, default) if isinstance(v, (pd.Series, pd.DataFrame)): v = v.iloc[0] if len(v) else default return v if v is not None else default

Bug Fix 2 — Geographic Bar Chart Column Error

Old code (broke on pandas ≥ 2.0): state_counts = ( filtered["state"].value_counts().reset_index() .rename(columns={"index": "State", "state": "Partners"}) ) px.bar(..., x="state", y="Partners") # Wrong → KeyError on pandas ≥ 2.0 Fix applied: _vc = filtered["state"].value_counts().reset_index() if "count" in _vc.columns: # pandas ≥ 2.0 layout state_counts = _vc.rename(columns={"state": "State", "count": "Partners"}) else: # pandas < 2.0 layout state_counts = _vc.rename(columns={"index": "State", "state": "Partners"}) px.bar(..., x="State", y="Partners") # Correct for both versions
